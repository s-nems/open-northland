# Fidelity — is the rebuild *faithful*, not just self-consistent?

Vinland's goal is a **faithful** rebuild of *Cultures – 8th Wonder* that can *then* be modded and
improved. "Faithful" is a different axis from "correct" in the test sense, and that difference is the
whole reason this file exists.

- **`npm test` proves self-consistency + determinism.** The pyramid (determinism, invariants — goods
  conservation, no-deadlock, path validity — golden traces) runs against the **synthetic fixture**. It
  proves the sim is reproducible and internally lawful. It does **not** prove the sim behaves like the
  original game. An agent can build a fully-green, fully-deterministic economy that plays nothing like
  *Cultures*, and no test would fire.
- **This file tracks the other axis: does each mechanic match the original?** It is a conformance
  ledger, maintained by `/iterate` (the per-step `fidelityBasis`) and `/reflect` (audits + tending).

## Why there is no automatic mechanics oracle

Fidelity is *partly* free and *mostly* not (see `docs/SOURCES.md`):

| Layer | Oracle? | How fidelity is pinned |
|---|---|---|
| **Data params** (recipes, ranges, timings, worker counts, atomic vocab) | **Yes — by construction** | Extracted from the original `.ini`/`.cif`; the pipeline verifies record counts (65 goods, 55 jobs, 105 weapons, …). Faithful as long as extraction is faithful. |
| **Assets** (`.pcx`/`.bmd` → PNG/atlas) | **Yes — OpenVikings pixel-oracle** | OpenVikings renders the originals; diff decoded output pixel-for-pixel + decoder round-trip tests. (Pending an owned game copy + the oracle.) |
| **Sim behavior** (atomic planner, economy loops, AI, pathing, combat, atomic timings) | **No** | OpenVikings' logic tick is a stub counter; the behavior is in neither the data nor the reverse-engineering. It can only be pinned by (a) the data-derived parameters, (b) the readable `.ini` *semantics*, (c) **calibration-by-observation** against the running original. Because nothing automatic catches drift here, **it must be tracked in this file.** |

## The faithful-first rule

A mechanic must match the original's behavior, pinned to one of the sources above. **Conscious
deviations are deferred and recorded — never the default.** If you intentionally diverge (a bug-fix, a
quality-of-life change, a simplification), log it under *Deviations* with the reason, so the faithful
baseline stays knowable and a future "faithful mode" / mod toggle remains possible. The content-is-data
architecture exists precisely so fidelity lives in data, not hard-coded systems.

## Conformance ledger

Status: `not-started` · `approximated` (running but not pinned to the original) · `faithful` (matches a
named source). Update the relevant row when a mechanic lands or is calibrated.

## Deviations (conscious divergences from the original)

Format: `- <mechanic>: <how it differs> — <why> (<commit>)`.

- Hunger rise: `needsSystem` raises hunger at a flat `HUNGER_RISE_PER_TICK`=ONE/4096, but the original
  drains hunger via per-activity-animation events (`event 30 2 -100`) and restores it per `eat_slot_food`
  event (`+4000`) on a large integer scale — an event-driven model that varies by which animation runs.
  The faithful model is hunger ticked by each completed atomic's `event (type=2, value)` against a
  ~10000-scale bar, deferred until the atomic `event (type,value)` vocabulary is decoded (a named
  Phase-1 extraction risk). The constant rate is the basic, bounded "hunger grows, eating resets it"
  core — strictly the right shape, with the per-activity rates as the recorded faithful target.

- Eat drive: the planner triggers eating at a flat `HUNGER_EAT_THRESHOLD`=¾·ONE and identifies food by
  the good's `food` id prefix, but the original drives eating off the per-animation hunger events
  (`event 30 2 <delta>` against a ~10000-scale bar — no single readable "eat at X" threshold) and maps
  the food goods to the eat slot below the readable rule files (no `iseatable` flag in `goodtypes.ini`).
  The eat *atomic id* (10) IS pinned to the original `setatomic` bindings; only the trigger level and
  the food-good identification are inferred, deferred until the atomic `event (type,value)` vocabulary
  is decoded (the same named Phase-1 extraction the hunger-rise deviation waits on) and a content
  eatable-flag/slot-binding is extracted. Strictly the right "eat when hungry, food restores it" shape.

- Fatigue rise: `needsSystem` raises fatigue at a flat `FATIGUE_RISE_PER_TICK`=ONE/8192, but the
  original drains rest via per-activity-animation events and restores it per `sleep` event
  (`event <at> 1 +4000`) on the same ~10000-scale bar hunger uses (type 1 = rest, type 2 = hunger).
  The faithful model is fatigue ticked by each completed atomic's `event (type=1, value)`, deferred to
  the same atomic `event (type,value)` decode the hunger deviation waits on. The reset *atomic id* (8,
  `sleep`) IS pinned to the original `setatomic` bindings; the half-of-hunger rise rate is an unpinned
  cadence stand-in (calibration-by-observation pending). The sleep DRIVE (planner choosing `sleep` at
  a fatigue threshold) has now landed — mirroring the eat drive, but the settler sleeps **in place**
  (the original sleeps at home; the housing/home system that would give a sleep target is a later
  slice) and the `FATIGUE_SLEEP_THRESHOLD`=¾·ONE trigger is unpinned like the eat trigger. Strictly
  the right "tired-then-rest" shape.

- Piety rise: `needsSystem` raises piety at a flat `PIETY_RISE_PER_TICK`=ONE/16384, but the original
  drains devotion via per-activity-animation events and restores it per `pray` event on a numbered
  channel (as type 1 = rest, type 2 = hunger; the religious channel is another id), on the same
  ~10000-scale bar. The faithful model is piety ticked by each completed atomic's `event (type,value)`,
  deferred to the same atomic `event (type,value)` decode the hunger/fatigue deviations wait on. The
  reset *atomic id* (12, `pray`) IS pinned to the original `setatomic 6 12 "..._pray"` bindings; the
  half-of-fatigue rise rate is an unpinned cadence stand-in (calibration-by-observation pending). The
  pray DRIVE (planner choosing `pray` at a piety threshold) **has now landed** — mirroring the eat/sleep
  drives, but as the first **target-bound** need it does the genuinely-new need→satisfier→**temple-target**
  lookup: it finds the nearest **temple** (a `kind === 'workplace'` building with no recipe/workers/stock
  — the structural signature of the original's `HOUSE_TYPE_WORK_TEMPLE` logictype 37, since the
  building→pray-need link isn't a readable flag, inferred like food→eat-slot), walks the settler there, and
  prays on arrival; the `PIETY_PRAY_THRESHOLD`=¾·ONE trigger is unpinned like the eat/sleep ones, and the
  temple-inference refines if a content building→need binding is later extracted. Strictly the right
  "devotion-lapses-then-pray-at-a-temple" shape. (The leisure resets `enjoy` id 17 / `make_love` id 78
  both restore the same `enjoyment` bar, in place — not target-bound like pray; see the Enjoyment note.)

- Enjoyment rise: `needsSystem` raises enjoyment at a flat `ENJOYMENT_RISE_PER_TICK`=ONE/32768, but the
  original drains leisure via per-activity-animation events and restores it per `enjoy` event
  (`event <at> 3 <delta>`, channel 3) on the same ~10000-scale bar the other needs use. The faithful
  model is enjoyment ticked by each completed atomic's `event (type=3, value)`, deferred to the same
  atomic `event (type,value)` decode the other need deviations wait on. The reset *atomic id* (17,
  `enjoy`) IS pinned to the original `setatomic 6 17 "..._civilist_enjoy"` bindings; the half-of-piety
  rise rate is an unpinned cadence stand-in (calibration-by-observation pending). The enjoy **DRIVE** is
  **deferred, not just unpinned**: unlike pray (satisfied at a temple, inferred from the
  no-recipe/no-worker/no-stock structural signature), `enjoy` has **no readable building satisfier** —
  `houses.ini` carries only the temple (lt 37) and a decorative wall (`work murek`, lt 55) in that
  signature class, so structural inference can't name a leisure site; the satisfier→need binding lives
  below the readable rule files and needs a content building→need extraction before a drive can land. So
  only the rise + reset are wired (no planner branch chooses `enjoy` yet). Strictly the right
  "leisure-lapses-then-have-fun" shape for the rise/reset half. The `make_love` atomic (id 78) is a
  **second reset of this same leisure bar**, not a separate need: its animation restores channel 3 too
  (`event <at> 3 +800`, a bigger boost than `enjoy`'s `+100`), so its `AtomicEffect` zeroes the same

- Experience threshold (`needfor*`): `experienceRequirementMet` reads a `need` requirement as "the sum
  of the settler's accrued XP across the line's `experienceTypes` ≥ `amount`", but the original measures
  the threshold per the same below-the-`.ini` XP logic the per-animation `event` deltas live in — for a
  two-`expType` line, whether it means "sum both" or "either alone" has no readable oracle. Summing the
  named tracks is the deterministic reading (a single threshold over the relevant specializations); the
  threshold `amount` and the track keying ARE pinned (the join key is the same `humanjobexperiencetypes`
  typeId the accrual writes), only the multi-track composition is inferred. Refine when the original's XP
  curve is observed. (No `baseRepeatCounter`→tier curve is interpreted yet — also deferred.)

- Production per-cycle ticks: the building-type `recipe.ticks` is pinned to ONE **reference tribe**

  coiner=200 vs frank coiner=60) and per output good. The merged building-type recipe carries a single
  `ticks`, so the per-tribe / per-output spread is collapsed to the reference tribe's primary-output
  length. The fully-faithful model is a per-tribe (and ideally per-output) recipe-timing table, deferred
  until there is tribe context at the sim/economy layer — strictly more faithful than the prior flat
  `DEFAULT_RECIPE_TICKS`=20 constant, which carried no source at all.

- Reproduction (birth rate + anchor + sex): `reproductionSystem` grows a tribe by **one settler per
  tick** while it has spare housing, births each newborn at the tribe's **lowest-id built `home`** tile,
  and gives it the `baby_female` sex. All three are unpinned: the original's birth cadence, family model,
  where a child appears, and which sex live below the readable `.ini` — there is no birth-rate key in
  `houses.ini`/`tribetypes.ini`, and the `make_love` atomic restores the leisure channel (id 78 →
  channel 3, like `enjoy`), it carries no birth yield. The newborn's **age class** (that it is a *baby*,
  not an adult trade) IS now pinned — `NEWBORN_AGE_CLASS` = `baby_female` (id 1) from `logicdefines.inc`
  `JOB_TYPE_HUMAN_BABY_FEMALE` + the `jobtypes.ini` records (`systems/ageclass.ts`); only *which sex* is
  the deterministic stand-in (the youngest age class, lowest id). The faithful *shape* — "with room to
  live, the colony grows toward the housing ceiling and stops there, as babies" — IS matched (the cadence
  is the gate: deterministic, self-limiting at `housingCapacity`, so the `populationWithinHousing`
  invariant can't be breached). The rate, the per-home anchor, and the sex remain the recorded faithful
  target, deferred until an oracle or calibration-by-observation against the running original. Strictly
  the right "grow into your housing, born young" shape, with no birth-rate constant invented.

- Growth cadence (baby→child→adult): **now LANDED** — `growthSystem` (`systems/ageclass.ts`) matures a
  born settler over `GROWUP_TICKS`=8192 per stage, sex preserved, removing the `Age` component once it is
  adult-eligible (see the GrowthSystem table row). The *structure* (age-class ids + female/male split) is
  faithful; only the **cadence** `GROWUP_TICKS` is the approximated, unpinned constant (no readable
  grow-up-rate key) — the established hunger-rise-style stand-in, calibration-by-observation pending. The
  **earlier known boundary is now closed:** the AI planner skips a still-growing settler (keyed on the
  `Age` component), so a baby/child no longer runs the adult eat/sleep/pray drives — faithful to "a baby
  is cared for, it doesn't self-feed". The `make_love`-driven family/pairing model (which settlers pair,
  whose child it is) remains a separate deferred mechanic.

- Carrier→vehicle pairing (cart logistics): **a conscious, recorded deferral — no per-carrier vehicle
  entity is modelled.** A carrier hauls abstractly at its tribe's best UNLOCKED carry capacity
  (`carrierCarryCapacity`: the largest `stockSlots` over the vehicles `jobEnablesVehicle` has unlocked,
  floor 1 on foot) — the *capacity number* and the *unlock* are both pinned to data, but the original's
  cart logistics are NOT: it assigns a specific vehicle per haul, a carrier visibly fetches/parks a cart
  at a vehicle store, and a vehicle's `logicgood` allow-list (which goods that cart may carry — each
  vehicle lists a per-type subset of good ids; a ship's set differs from a handcart's) constrains the
  haul. **This is oracle-blocked**:
  `vehicletypes.ini` carries only the static type table (`type`/`name`/`logicsize`/`stockslots`/the
  `logicgood`+`logicpassenger` allow-lists/`debug*` cosmetics) — there is **no** carrier-job→vehicle
  binding, no dispatch rule, no "fetch a cart" key; that behavior lives in the engine logic below the
  readable `.ini`, and OpenVikings' sim is a stub (no logistics to diff against). So modelling a
  per-carrier vehicle entity now would be *invented*, not faithful. Two specific sub-divergences are
  folded in: (a) the `logicgood` allow-list is **not applied** — `carrierCarryCapacity` takes the best
  `stockSlots` regardless of whether the unlocked cart may carry the good being hauled (a ship's 200-slot
  capacity would apply even to a good no vehicle's allow-list permits); the `VehicleType` IR does not even
  extract `logicgood` today (only `typeId`/`name`/`stockSlots`/`passengerSlots`/`logicSize`), so applying
  the filter would require a new extraction first — deferred with the cart entity; (b) the carrier
  *dispatch* (which
  carrier takes which haul) is the nearest-Manhattan stand-in, and "never deliver back into the producer"
  is our unpinned choice. The faithful *shape* — "a carrier moves a batch of finished goods to a store,
  bigger as the tribe unlocks bigger carts" — IS matched; the cart-as-entity, the per-vehicle goods
  filter, and the dispatch policy are the recorded faithful target, deferred to a dedicated vehicle-entity
  slice once an oracle exists (calibration-by-observation against the running original, or a future
  OpenVikings logistics decode). This closes the JobSystem's last unmodelled Phase-3 behavior as a
  *decision*, not a stub: the data is consumed, the divergence is knowable, the faithful path is named.
