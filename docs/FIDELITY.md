# Fidelity ‚Äî is the rebuild *faithful*, not just self-consistent?

Vinland's goal is a **faithful** rebuild of *Cultures ‚Äì 8th Wonder* that can *then* be modded and
improved. "Faithful" is a different axis from "correct" in the test sense, and that difference is the
whole reason this file exists.

- **`npm test` proves self-consistency + determinism.** The pyramid (determinism, invariants ‚Äî goods
  conservation, no-deadlock, path validity ‚Äî golden traces) runs against the **synthetic fixture**. It
  proves the sim is reproducible and internally lawful. It does **not** prove the sim behaves like the
  original game. An agent can build a fully-green, fully-deterministic economy that plays nothing like
  *Cultures*, and no test would fire.
- **This file tracks the other axis: does each mechanic match the original?** It is a conformance
  ledger, maintained by `/iterate` (the per-step `fidelityBasis`) and `/reflect` (audits + tending).

## Why there is no automatic mechanics oracle

Fidelity is *partly* free and *mostly* not (see `docs/SOURCES.md`):

| Layer | Oracle? | How fidelity is pinned |
|---|---|---|
| **Data params** (recipes, ranges, timings, worker counts, atomic vocab) | **Yes ‚Äî by construction** | Extracted from the original `.ini`/`.cif`; the pipeline verifies record counts (65 goods, 55 jobs, 105 weapons, ‚Ä¶). Faithful as long as extraction is faithful. |
| **Assets** (`.pcx`/`.bmd` ‚Üí PNG/atlas) | **Yes ‚Äî OpenVikings pixel-oracle** | OpenVikings renders the originals; diff decoded output pixel-for-pixel + decoder round-trip tests. (Pending an owned game copy + the oracle.) |
| **Sim behavior** (atomic planner, economy loops, AI, pathing, combat, atomic timings) | **No** | OpenVikings' logic tick is a stub counter; the behavior is in neither the data nor the reverse-engineering. It can only be pinned by (a) the data-derived parameters, (b) the readable `.ini` *semantics*, (c) **calibration-by-observation** against the running original. Because nothing automatic catches drift here, **it must be tracked in this file.** |

## The faithful-first rule

A mechanic must match the original's behavior, pinned to one of the sources above. **Conscious
deviations are deferred and recorded ‚Äî never the default.** If you intentionally diverge (a bug-fix, a
quality-of-life change, a simplification), log it under *Deviations* with the reason, so the faithful
baseline stays knowable and a future "faithful mode" / mod toggle remains possible. The content-is-data
architecture exists precisely so fidelity lives in data, not hard-coded systems.

## Conformance ledger

Status: `not-started` ¬∑ `approximated` (running but not pinned to the original) ¬∑ `faithful` (matches a
named source). Update the relevant row when a mechanic lands or is calibrated.

**Keep each row to one line: `status` + how it's pinned (the source file + the *one* thing that could
be wrong) + what stays `approximated` and why (the deferred pointer).** The blow-by-blow ‚Äî record
counts, id enumerations, "0 dangling", the hands-on transcript ‚Äî belongs in the commit message, not
the cell; a row that re-narrates its implementation has stopped being a ledger. Extend a row in place;
don't append a fresh "and now X also lands" clause each iteration.

  where a child appears, and which sex live below the readable `.ini` ‚Äî there is no birth-rate key in
  `houses.ini`/`tribetypes.ini`, and the `make_love` atomic restores the leisure channel (id 78 ‚Üí
  channel 3, like `enjoy`), it carries no birth yield. The newborn's **age class** (that it is a *baby*,
  not an adult trade) IS now pinned ‚Äî `NEWBORN_AGE_CLASS` = `baby_female` (id 1) from `logicdefines.inc`
  `JOB_TYPE_HUMAN_BABY_FEMALE` + the `jobtypes.ini` records (`systems/lifecycle/ageclass.ts`); only *which sex* is
  the deterministic stand-in (the youngest age class, lowest id). The faithful *shape* ‚Äî "with room to
  live, the colony grows toward the housing ceiling and stops there, as babies" ‚Äî IS matched (the cadence
  is the gate: deterministic, self-limiting at `housingCapacity`, so the `populationWithinHousing`
  invariant can't be breached). The rate, the per-home anchor, and the sex remain the recorded faithful
  target, deferred until an oracle or calibration-by-observation against the running original. Strictly
  the right "grow into your housing, born young" shape, with no birth-rate constant invented.

- Growth cadence (baby‚Üíchild‚Üíadult): **now LANDED** ‚Äî `growthSystem` (`systems/lifecycle/ageclass.ts`) matures a
  born settler over `GROWUP_TICKS`=8192 per stage, sex preserved, removing the `Age` component once it is
  adult-eligible (see the GrowthSystem table row). The *structure* (age-class ids + female/male split) is
  faithful; only the **cadence** `GROWUP_TICKS` is the approximated, unpinned constant (no readable
  grow-up-rate key) ‚Äî the established hunger-rise-style stand-in, calibration-by-observation pending. The
  **earlier known boundary is now closed:** the AI planner skips a still-growing settler (keyed on the
  `Age` component), so a baby/child no longer runs the adult eat/sleep/pray drives ‚Äî faithful to "a baby
  is cared for, it doesn't self-feed". The `make_love`-driven family/pairing model (which settlers pair,
  whose child it is) remains a separate deferred mechanic.

- Carrier‚Üívehicle pairing (cart logistics): **a conscious, recorded deferral ‚Äî no per-carrier vehicle
  entity is modelled.** A carrier hauls abstractly at its tribe's best UNLOCKED carry capacity
  (`carrierCarryCapacity`: the largest `stockSlots` over the vehicles `jobEnablesVehicle` has unlocked,
  floor 1 on foot) ‚Äî the *capacity number* and the *unlock* are both pinned to data, but the original's
  cart logistics are NOT: it assigns a specific vehicle per haul, a carrier visibly fetches/parks a cart
  at a vehicle store, and a vehicle's `logicgood` allow-list (which goods that cart may carry ‚Äî each
  vehicle lists a per-type subset of good ids; a ship's set differs from a handcart's) constrains the
  haul. **This is oracle-blocked**:
  `vehicletypes.ini` carries only the static type table (`type`/`name`/`logicsize`/`stockslots`/the
  `logicgood`+`logicpassenger` allow-lists/`debug*` cosmetics) ‚Äî there is **no** carrier-job‚Üívehicle
  binding, no dispatch rule, no "fetch a cart" key; that behavior lives in the engine logic below the
  readable `.ini`, and OpenVikings' sim is a stub (no logistics to diff against). So modelling a
  per-carrier vehicle entity now would be *invented*, not faithful. Two specific sub-divergences are
  folded in: (a) the `logicgood` allow-list is now applied **on the boat-as-mobile-store DEPOSIT side** ‚Äî
  `stockCapacity` (`systems/shared.ts`) gates a haul INTO a placed `Vehicle` hull by `vehicleMayCarry` +
  `stockSlots` (the *Boat hull* row's cargo-load gate), so a forbidden good is refused at the deposit
  seam ‚Äî but it is **still not applied on the carrier CARRY-CAPACITY side**: `carrierCarryCapacity` takes
  the best `stockSlots` over the unlocked vehicles regardless of whether that cart's `cargoGoods` permits
  the good being hauled (a ship's 200-slot capacity would size a batch even for a good no cart's allow-list
  permits). The deposit gate is the WHAT-a-hold-accepts half; wiring the filter into the carry-BATCH sizing
  rides with the per-carrier cart entity (the data is ready, the dispatch is the blocker); (b) the carrier *dispatch* (which
  carrier takes which haul) is the nearest-Manhattan stand-in, and "never deliver back into the producer"
  is our unpinned choice. The faithful *shape* ‚Äî "a carrier moves a batch of finished goods to a store,
  bigger as the tribe unlocks bigger carts" ‚Äî IS matched; the cart-as-entity, the per-vehicle goods
  filter, and the dispatch policy are the recorded faithful target, deferred to a dedicated vehicle-entity
  slice once an oracle exists (calibration-by-observation against the running original, or a future
  OpenVikings logistics decode). This closes the JobSystem's last unmodelled Phase-3 behavior as a
  *decision*, not a stub: the data is consumed, the divergence is knowable, the faithful path is named.

- Per-tribe construction cost: **a conscious, recorded collapse ‚Äî `BuildingType.construction` holds one
  cost per `typeId`, not per `(tribe, typeId)`.** The graphics table (`DataCnmd/budynki12/houses/houses.ini`,
  `[GfxHouse]`) keys each `LogicConstructionGoods` line by a `LogicTribeType`, and the SAME logic `typeId`
  carries a genuinely DIFFERENT build cost per civilization: viking/frank/byzantine model a residence as an
  upgrade *chain* (typeId 6 = home level 4 = `27 27`, ornaments only ‚Äî the incremental tier cost), while
  egypt/saracen model that same typeId as a standalone *full* build (the cumulative `4 5 5 2 ‚Ä¶ 27 27` list).
  `extractConstructionCosts` collapses this to the **lowest-`LogicTribeType`** record (the deterministic
  "reference tribe" convention `fillBuildingRecipes` already uses for `recipe.ticks`), so a leveled `home`
  reads its OWN tier's upgrade cost ‚Äî coherent, but it loses the per-tribe spread (a saracen residence costs
  the viking chain's tier cost in the IR, not its own cumulative figure). **A second collapse rides along:**
  within ONE `[GfxHouse]` record a `typeId` can map to more than one `sizeIdx` (e.g. "viking pottery"
  `LogicType {1:21, 2:21}`, and a multi-stage wonder repeats one typeId across rising sizes 0,1,2 with
  cumulative costs `5 5 5 ‚Üí 6 6 5 5 5 ‚Üí 7 7 6 6 5 5 5`); the **lowest `sizeIdx`** (the base/first build stage)
  wins, so a wonder reads its first-stage cost, not the cumulative final. Both are **not** oracle-blocked (the
  per-tribe / per-stage data is fully readable); they are deliberate shape choices to keep `construction` a
  single flat field until a system consumes the richer key. The fully-faithful model is a per-`(tribe, typeId,
  sizeIdx)` construction table ‚Äî the same kind of per-tribe deferral the `recipe` table carries (the
  *Production per-cycle ticks* row, axis (a)) ‚Äî landed together when a tribe/stage-aware ConstructionSystem
  needs it.

- Settler facing (bob direction layout): the `CR_Hum_Body` bobs do **NOT** lay their 8 directions out as a
  uniform screen-angle rotation, so no `atan2`-bucketing formula can pick them. The true per-block facing
  (a human read off each labelled direction block) is `0 SW, 1 W, 2 NW, 3 NE, 4 E, 5 SE, 6 S, 7 N`.
  `readFacing` (`packages/render/src/data/scene.ts`) therefore maps the settler's grid step **directly** to the
  block facing that step's iso-screen heading via the `STEP_TO_FACING` table (`E‚ÜíSE‚Üí5, S‚ÜíSW‚Üí0, W‚ÜíNW‚Üí2,
  N‚ÜíNE‚Üí3`, plus the four diagonal steps), with no angle/rounding. **Earlier this used a screen-angle
  formula assuming a regular rotation** ‚Äî it happened to get the SE heading right but rendered other
  headings facing the wrong way (e.g. a settler walking screen-up-left faced down-left). The direct table
  is a faithful match to the asset, not an approximation; `DEFAULT_FACING` is block 5 (SE, toward camera).
  (settler facing ‚Äî direct grid-step‚Üíblock table)

- Movement / facing granularity ‚Äî **8-connected movement + full 8-facing gait (LANDED)**: pathfinding is
  now 8-connected ‚Äî `packages/sim/src/nav/terrain.ts` `steps()` emits diagonal steps alongside the
  orthogonal `NEIGHBOUR_OFFSETS`, at ‚àö2 cost under the octile heuristic, with a no-corner-cut rule (see
  the *A\* pathfinding* row). Because a path leg can now be a **diagonal grid step**, the render's
  `readFacing` (`packages/render/src/data/scene.ts`, `STEP_TO_FACING`) produces the four screen-cardinal
  facings a diagonal maps to (W/E/S/N = blocks 1/4/6/7) as well as the four screen-diagonal facings a
  single-axis step maps to (SW/NW/NE/SE = blocks 0/2/3/5) ‚Äî so **all eight** `CR_Hum_Body` directions are
  now reachable in both the empty walk and the loaded carry gait (the old 4-facing cap is gone). The
  per-axis `MovementSystem` already walks a diagonal waypoint on both axes, so no movement change was
  needed. **APPROXIMATED (behavior, no oracle):** whether the REAL engine moves on an 8-connected graph
  (vs finer/continuous), and its exact diagonal cost + corner policy, are unpinned ‚Äî the original's
  humans visibly walk 8 directions, which motivates the choice, but OpenVikings' sim is a stub. Per-
  DIRECTION screen speed is **not** normalised: `stepToward` keeps every cell-step at the same tick cost
  (a diagonal covers ‚àö2 tiles in the same ticks), and since the iso projection makes tile‚Üíscreen speed
  anisotropic anyway, uniform screen speed would be a separate calibration-by-observation, not attempted
  here. (8-connected movement + 8-facing gait)

- RTS unit orders (select + box-select + move for civilians): the player can rubber-band-select the
  human player's units and right-click to send them anywhere (`moveUnit` command ‚Üí `MoveGoal`), the
  standard-RTS control scheme. The ORIGINAL *Cultures* does NOT box-select-and-move civilian settlers ‚Äî
  they are autonomous, steered indirectly (the "hand" picks up/drops a settler; professions are assigned;
  soldiers get direct orders). This RTS-style direct control is a **conscious quality-of-life deviation**,
  requested for the rebuild. It is kept faithful *in spirit* by making the order a SOFT, TIMED override,
  not a seizure (see below): the economy AI reclaims the unit after a hold, so autonomy ‚Äî the soul of
  Cultures ‚Äî is never disabled, only briefly deferred. The `Owner{player}` ownership model + the 16-player
  cap (`components/ownership.ts`) that gate "whose units you may command" are new scaffolding for the
  ROADMAP's up-to-N-players target, not an original data structure. (unit selection + move orders)

- Player move-order dwell (`MOVE_ORDER_HOLD_CIVILIAN`=50, `MOVE_ORDER_HOLD_SOLDIER`=300 ticks;
  `systems/conflict/orders.ts`): after a moved unit ARRIVES it stands `holdTicks` before the economy AI
  re-tasks it ‚Äî short for a worker (it returns to its job), long for a combatant (it holds position),
  chosen by whether the unit carries a `Health`/`Weapon` (combatant). The needs drives (eat/sleep/pray)
  still preempt the hold, so a warrior may wander off to eat ‚Äî the faithful "he'll soon go do his own

  the readable `.ini` and OpenVikings' sim is a stub. The interrupt-on-order (a move cancels the current
  atomic) and the per-player check being absent (any owned unit is orderable ‚Äî no issuing-player field
  until lockstep) are the other unpinned choices. Inert in the goldens (no `moveUnit` in the golden
  stream; `PlayerOrder` is a separate optional component ‚Üí golden hash untouched). (move-order dwell)

- Idle-spacing / soft collision (`deStackIdle` in `systems/conflict/ai.ts`, `SPACING_SEARCH_CAP`=48):
  settlers do NOT hard-collide ‚Äî a walker passes freely through any occupied tile (so a crowd can thread
  the gaps between village buildings without deadlocking) ‚Äî but they will not come to REST stacked on one
  another. When an owned settler has ARRIVED with nothing left to do (every economic + needs + player-order
  drive above it declined) and shares its tile with a *lower-id* resting owned settler, it steps to the
  nearest free, unblocked cell (a canonical breadth-first ring search); the lowest-id occupant is the
  keeper and holds its ground, so a stack of N spreads to N distinct tiles and then stops (no churn).
  **Faithful in spirit:** the original engine gives every landscape type a per-cell `maximumValency` (the
  extracted `maxValency`, already on the terrain graph) ‚Äî units cluster only up to a cell's capacity ‚Äî so
  "don't rest stacked" is the original's model, not an invention. **Approximated (no oracle):** the target
  is *one settler per resting cell* (not the type's real valency number ‚Äî the readable data gives the cap
  but not how the engine distributes overflow), the de-stack is a last-resort planner drive rather than a
  reservation held during pathing, and the "reluctant to ENTER an occupied tile" pathfinding penalty the
  user also described is deferred (the de-stack at rest already delivers the visible result ‚Äî units don't
  pile up ‚Äî so A* stays unweighted; a soft occupancy cost is a later refinement, docs/TECH-DEBT.md). Gated
  on `Owner`, so it only ever moves gameplay units; the unowned golden/economy fixtures build an empty
  occupancy set and are byte-identical. Group move-orders reach distinct tiles via a client-side formation
  (app `view/picking.ts` `formationTiles`), with this de-stack as the sim-side safety net. (idle spacing)
